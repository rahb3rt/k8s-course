D1

Substrate & the Control Plane

Complete

Start below Kubernetes and climb up: a container built by hand from Linux primitives, then a control plane assembled from four files on disk, all the way down to the protobuf bytes in etcd — and finally, killing it to watch what "the cluster is down" really means.

Labs run

1.1
A container, by hand
unshare a process into its own PID / net / uts / mnt namespaces — no Docker. Proved the isolation, then found the missing third leg: rootfs.
1.2
kubeadm bootstrap & dissection
Init the control plane, then read every static pod manifest, walk the PKI tree, and join the worker via the TLS bootstrap-token protocol.
1.3
Read etcd raw
Query keys straight out of etcd with etcdctl and see objects stored as protobuf — proving the API server is etcd's sole interpreter.
☂ 1.4
Breakage — kill etcd
Move etcd.yaml out of the manifests dir; watch the API server's /readyz hit 500 and crash-loop. Diagnose with kubectl dead; recover by restoring one file. Zero data lost.

Key commands

# Build the kernel core of a container — one command, no daemon
sudo unshare --pid --fork --mount-proc --uts --net --ipc --mount bash

# The control plane is four files the kubelet babysits
ls /etc/kubernetes/manifests/   # etcd · apiserver · controller-manager · scheduler

# Read cluster state straight from etcd (etcdctl lives in the distroless image)
kubectl -n kube-system exec -i etcd-lima-cp -- etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  get /registry --prefix --keys-only

# When kubectl is dead, drop a layer (node-local!)
sudo crictl ps                      # containers on THIS node
sudo journalctl -u kubelet -f       # the kubelet's live view

Mental models locked in

A container is three Linux primitives.

Namespaces (what it sees) + cgroups (what it can use) + a pivoted rootfs (its files). Every "weird pod" bug bottoms out in one of these.

The control plane bootstraps from disk.

Static pods in /etc/kubernetes/manifests/ run without a scheduler or API server. hostNetwork: true is why they start before any CNI exists — CoreDNS had to wait; etcd never did.

etcd is truth; the API server is its only interpreter.

Objects are stored as protobuf. A 40-line manifest becomes kilobytes of status, QoS, digests, and managedFields. "Everything goes through the apiserver" is architecture, not etiquette.

Control plane down ≠ workloads down.

etcd, apiserver, and kubectl were all dead — the nginx pods served traffic the whole time. Different sev levels. The kubelet runs pods from local state.

Read the error's layer.

refused = the process isn't listening; timeout = it's there but stuck on a dependency. Data outlives process: /var/lib/etcd survived, so recovery was lossless.

D2

Networking & the Dataplane

Complete

How a 10.244.x.x packet crosses a 192.168.104.0/24 link the VMs don't route pod IPs on — traced end to end with tcpdump, then broken and fixed by hand.

Labs run

2.1
Map the dataplane
Found cni0 (bridge), flannel.1 (VXLAN), and a veth per pod. The routing table makes the same-node (dev cni0) vs cross-node (dev flannel.1) decision.
2.2
Trace the packet, two layers
Caught one ICMP echo in two envelopes: inner pod→pod on flannel.1, outer node→node UDP/8472 on eth0. VXLAN encapsulation, live.
2.3
A Service is just rules
Walked KUBE-SERVICES → KUBE-SVC → KUBE-SEP: a ClusterIP is a NAT match target, load-balanced by a statistic random probability chain, DNAT'd to a pod. No proxy in the path.
☂ 2.4
Breakage — the vanishing replies ✓ solved solo
A DROP on inbound UDP/8472 killed cross-node traffic. Localized by hand to the netfilter INPUT chain — reply seen on eth0, gone before flannel.1. Found the rule; DNS restored.
2.5
The firewall that isn't one
A default-deny NetworkPolicy did nothing — Flannel doesn't enforce policy. Intent stored in etcd with no enforcer to realize it.

The debugging method — turn the crank

Bisection for "X can't reach Y"
  1. Reproduce — see the failure yourself before touching anything.
  2. Narrow — change one variable to halve it: all traffic or one port? cross-node or same-node? one pod or all?
  3. Walk the path — follow the packet hop by hop with tcpdump; find where it stops.
  4. Inspect at the break — routes, iptables, firewall — wherever step 3 pointed.
  5. Fix & verify the original symptom — not a proxy metric; the thing the user reported.

Key commands

# The routing table IS the same-node vs cross-node decision
ip route          # 10.244.1.0/24 dev cni0  |  10.244.0.0/24 ... dev flannel.1

# See both envelopes of a cross-node hop
sudo tcpdump -ni eth0 udp port 8472     # outer: node -> node (VXLAN)
sudo tcpdump -ni flannel.1 icmp         # inner: pod  -> pod  (decapsulated)

# A Service is NAT rules kube-proxy writes
sudo iptables-save -t nat | grep <clusterIP>

# tcpdump taps BEFORE netfilter: on eth0 but not flannel.1 = dropped in INPUT
sudo iptables -L INPUT -n -v --line-numbers

Mental models locked in

Flat pod network, two implementations.

Every pod gets a real routable IP. Same-node hops cross the cni0 bridge; cross-node hops get VXLAN-encapsulated (node IPs outside, pod packet inside). Overhead = 50 bytes → pod MTU 1450.

A Service is a NAT illusion, not a process.

The ClusterIP exists only as an iptables match; DNAT rewrites it to a pod IP and conntrack pins the choice. Load balancing is a chain of probability rules. Off-network sources get masqueraded.

Response ≠ hang — and short names are slow.

An NXDOMAIN means DNS works; a timeout means the path is dead. ndots:5 + search domains make short names cost 4–5 lookups — prefer FQDNs.

The API stores intent; something must enforce it.

A NetworkPolicy under Flannel is inert — accepted, stored, unenforced, silently fails open. "Security object exists" ≠ "security is enforced." Verify with a policy CNI (Calico/Cilium) and a live deny test.

D3

Storage, Lifecycle & the OOM Killer

Complete

State and time: how pods are torn down, who the kernel kills under memory pressure, and how data outlives the pod that wrote it.

Labs run

3.1
How a pod dies
A pod that ignored SIGTERM hung in Terminating for the full 30s grace period, then SIGKILL. The Day-1 PID-1 signal trap, made real.
3.2
QoS → oom_score_adj
Read -997 / ~992 / 1000 off Guaranteed / Burstable / BestEffort pods — the exact integers the kernel OOM killer reads to pick a victim.
☂ 3.3
OOMKilled, live
A pod asked for 300Mi under a 128Mi limit → cgroup OOM → exit 137 (SIGKILL), confirmed in the node's dmesg.
3.4
Storage that survives
PVC Pending with no provisioner → installed local-path → WaitForFirstConsumer → bound on pod schedule. A file read back by a pod created after its writer was deleted.
3.5
StatefulSets: stable identity
Ordinal names (sts-0/1), one private PVC per pod, ordered boot. Deleted sts-0 came back as itself, reattached to its own volume, with stable per-pod DNS.

Key commands

# The grace period lives on the pod; a SIGTERM-ignoring PID 1 waits it out
kubectl get pod X -o jsonpath='{.spec.terminationGracePeriodSeconds}'

# QoS class, and the kernel kill-priority it maps to
kubectl get pods -o custom-columns='NAME:.metadata.name,QOS:.status.qosClass'
kubectl exec POD -- cat /proc/1/oom_score_adj   # -997 / ~992 / 1000

# The kernel's own record of an OOM kill (run on the node)
sudo dmesg | grep -i 'killed process'

# Why is my PVC stuck Pending? -> no provisioner, or WaitForFirstConsumer
kubectl get storageclass
kubectl describe pvc X | tail

Mental models locked in

Termination is a negotiation with a deadline.

SIGTERM → wait terminationGracePeriodSeconds → SIGKILL. Endpoint removal races the shutdown, so fast-exiting apps can drop traffic mid-rollout — a preStop sleep is the fix.

Two OOMs, two causes.

Per-container: blew your own limits.memory → deterministic cgroup kill, QoS-irrelevant. Node-level: node is full → kubelet evicts (then kernel kills) by QoS — BestEffort first. Set requests, or volunteer to die first.

A claim is inert without a provisioner.

Same pattern as NetworkPolicy: a PVC is intent; a StorageClass + provisioner is the actor. WaitForFirstConsumer delays binding until a pod schedules, so node-local volumes land on the right node.

Deployments are cattle; StatefulSets are named livestock.

Stable ordinal identity, one private PVC per pod (volumeClaimTemplates), ordered lifecycle, and per-pod DNS via a headless Service. Everything a database needs and a Deployment can't give.

Debugging toolkit

Reference

The fallback kit for when the friendly tools are gone. Grows as the course does.

kubectl hangs
apiserver up but stuck on a dependency (usually etcd). Check /readyz — a 500 names the failing subsystem.
connection refused
Nothing listening on the port — the process is down or in crash-loop backoff. Different layer than a timeout.
sudo crictl ps
Containers on this node only — crictl is node-local. Check the node that actually runs the pod. Needs root (socket is 0660 root:root).
journalctl -u kubelet -f
The kubelet's live narration. Housekeeping took too long = it's straining on API/etcd latency.
restart count
A scar that outlives the incident — evidence of how hard and how long a component flapped.
static pod won't die
The file is truth, not the API. Fix a broken control-plane pod by editing the manifest; bounce it by moving the file out and back.
cross-node broken, same-node OK
Suspect the overlay port — a firewall / security group dropping UDP 8472 (Flannel VXLAN), 4789 (std VXLAN), or 179 (Calico BGP). DNS times out too, since CoreDNS is often cross-node.
tcpdump vs iptables
tcpdump taps the wire before netfilter. A packet on eth0 but never on flannel.1 was dropped in the INPUT chain in between.
NetworkPolicy does nothing
The object is stored but unenforced unless the CNI implements policy — Flannel doesn't. Fails open, no error. Test-apply a deny and confirm traffic actually stops.
stuck in Terminating
PID 1 isn't exiting on SIGTERM; kubelet SIGKILLs at terminationGracePeriodSeconds. Fix: handle the signal, add an init shim, or tune the grace period.
OOMKilled / exit 137
Container exceeded its own limits.memory (cgroup OOM) — raise the limit or fix the leak. 137 = 128 + 9 (SIGKILL). Node-wide instead? Check QoS + evictions.
PVC stuck Pending
No StorageClass/provisioner, an empty storageClassName, or WaitForFirstConsumer waiting for a consuming pod. describe pvc names which.
