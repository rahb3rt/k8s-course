# Lab 1.4 — Break etcd, debug with kubectl dead, recover

**Goal:** feel a real control-plane outage and prove the data plane survives it.
Fully reversible — you stop etcd's *process*, never touch its data in `/var/lib/etcd`.

Run on **cp**. All commands here assume you're in a `limactl shell cp`.

## Phase 0 — Steady state (never break an unhealthy cluster)

```bash
# crictl needs its socket configured — set it up BEFORE the outage, you can't mid-crisis
sudo tee /etc/crictl.yaml >/dev/null <<'EOF'
runtime-endpoint: unix:///run/containerd/containerd.sock
image-endpoint: unix:///run/containerd/containerd.sock
timeout: 10
EOF

# A workload to test survival with (lands on the worker — cp is tainted)
kubectl create deployment web --image=nginx --replicas=2
kubectl rollout status deployment/web
kubectl get nodes && kubectl get pods -o wide     # baseline: all Ready/Running
```

In a **second** cp terminal, tail the kubelet: `sudo journalctl -u kubelet -f`

## Hypothesis (predict before injecting)
1. What happens to `kubectl`?
2. Do the running `web` pods keep serving?
3. Will the kubelet restart etcd on its own?

## Phase 1 — Inject

```bash
sudo crictl ps | grep -E 'etcd|apiserver'          # before
sudo mv /etc/kubernetes/manifests/etcd.yaml /root/etcd.yaml   # kubelet stops the static pod
```

## Phase 2 — Observe the blast radius

```bash
sudo crictl ps | grep etcd          # gone
kubectl get nodes                   # HANGS then times out (apiserver up, etcd unreachable)
# The workloads — check the WORKER (crictl is node-local; cp is tainted so web runs there):
limactl shell worker -- sudo crictl --runtime-endpoint unix:///run/containerd/containerd.sock ps | grep nginx
```

Kubelet log shows the whole cascade: `/readyz` HTTP **500** (apiserver alive, etcd down),
failing node-lease heartbeats, and `Failed deleting a mirror pod ... etcd-cp` (can't reach
the API to clean up). **The nginx pods keep serving the entire time.**

## Phase 3 — Diagnose (kubectl is useless; drop a layer)

```bash
sudo crictl ps -a | grep -E 'kube-apiserver|etcd'
sudo crictl logs $(sudo crictl ps -aq --name kube-apiserver | head -1) 2>&1 | tail -20  # etcd connection refused
ls -l /etc/kubernetes/manifests/    # SMOKING GUN: etcd.yaml is missing
```

## Phase 4 — Recover (self-heals)

```bash
sudo mv /root/etcd.yaml /etc/kubernetes/manifests/   # kubelet restarts etcd from the file
# Wait ~20-90s. The apiserver crash-looped during the outage and needs its backoff to expire.
sudo crictl ps | grep etcd
kubectl get nodes                    # answers again
kubectl get deployment web           # still 2/2 — ZERO data lost
```

If the apiserver stays `connection refused` for >2 min (long backoff), kick it:
```bash
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /root/ && sleep 3 && \
sudo mv /root/kube-apiserver.yaml /etc/kubernetes/manifests/
```

## Lessons
- **Control plane down ≠ workloads down.** The kubelet runs pods from local state.
- When kubectl is dead: `crictl` (node-local!), `journalctl -u kubelet`, the manifests dir.
- `refused` (nothing listening) vs `timeout` (stuck on a dependency) point at different layers.
- The etcd *process* died; its *data* in `/var/lib/etcd` persisted → lossless recovery.
  Losing the data dir is Day 4's restore drill.
