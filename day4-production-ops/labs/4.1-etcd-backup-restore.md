# Lab 4.1 — etcd backup & restore (resurrect a dead cluster)

The dark twin of Lab 1.4. There you stopped etcd's *process* and lost nothing. Here you
**destroy the data** and rebuild the cluster from a snapshot. Run on **cp**.

- **RPO** (Recovery Point Objective) = data you can lose = *time since last backup*. Anything
  written after the snapshot is gone on restore.
- **RTO** (Recovery Time Objective) = how long recovery takes. In this lab it was ~2 minutes.

> Architect's lens: single-node etcd is a SPOF (prod runs 3/5 members), but replication
> protects against node loss, **not** corruption or a bad write — those replicate instantly.
> So you need quorum **and** frequent verified snapshots **and** a *tested* restore runbook.

---

## Part A — Take a verified backup

```bash
# 1. Capture etcd's identity — needed for a correct restore
sudo grep -E 'name=|initial-cluster=|advertise-peer-urls=' /etc/kubernetes/manifests/etcd.yaml
#   --name=lima-cp
#   --initial-cluster=lima-cp=https://192.168.104.1:2380
#   --initial-advertise-peer-urls=https://192.168.104.1:2380

# 2. Snapshot (etcdctl lives in the distroless etcd pod; save into the mounted data dir)
kubectl -n kube-system exec -i etcd-lima-cp -- etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /var/lib/etcd/backup.db

# 3. Move it OUT of the data dir (we're about to wipe that) and VERIFY it
sudo cp /var/lib/etcd/backup.db /root/etcd-backup.db
kubectl -n kube-system exec -i etcd-lima-cp -- etcdutl snapshot status /var/lib/etcd/backup.db -w table
```
A verified snapshot shows hash / revision / total keys / size. An unverified backup is a prayer.

## Part B — Disaster + restore

```bash
# Install etcdutl on the HOST (the etcd pod is down during a restore — match your image tag)
ETCD_VER=v3.5.24
curl -sL https://github.com/etcd-io/etcd/releases/download/${ETCD_VER}/etcd-${ETCD_VER}-linux-arm64.tar.gz -o /tmp/etcd.tar.gz
sudo tar xzf /tmp/etcd.tar.gz -C /usr/local/bin --strip-components=1 \
  etcd-${ETCD_VER}-linux-arm64/etcdctl etcd-${ETCD_VER}-linux-arm64/etcdutl

date +%T                                              # start the RTO clock

# DISASTER: stop the whole control plane, then destroy the data
sudo mv /etc/kubernetes/manifests /etc/kubernetes/manifests.off   # kubelet stops static pods
sudo rm -rf /var/lib/etcd                                          # the brain is gone
kubectl get nodes                                                 # connection refused

# RESTORE: rebuild the data dir with the SAME identity (or the member won't match)
sudo etcdutl snapshot restore /root/etcd-backup.db \
  --name=lima-cp \
  --initial-cluster=lima-cp=https://192.168.104.1:2380 \
  --initial-advertise-peer-urls=https://192.168.104.1:2380 \
  --data-dir=/var/lib/etcd

# REVIVE: return the manifests; kubelet restarts etcd from the restored data
sudo mv /etc/kubernetes/manifests.off /etc/kubernetes/manifests
# Wait 30-90s. The apiserver crash-loops briefly against fresh etcd (backoff) — retry kubectl.

date +%T ; kubectl get nodes ; kubectl get cm         # RTO end; nodes Ready; canary check
```

## Verify (the canary test)
- `canary-before` (created **before** the snapshot) → **present**.
- `canary-after` (created **after** the snapshot) → **GONE.** That absence is your RPO.

## Gotchas
- Restore `--name` / `--initial-cluster` / `--initial-advertise-peer-urls` **must** match the
  running etcd's flags, or you get a cluster-ID / member mismatch and etcd won't start.
- `etcdutl` refuses to restore over an existing `--data-dir`; wipe it first.
- For a 3-node etcd, restore on **one** node with `--initial-cluster` listing all members, copy
  the data dir out, and bring the others up as new members — never restore all three independently.

See also `etcd-backup.sh` for a cron-able, timestamped, self-pruning backup.
