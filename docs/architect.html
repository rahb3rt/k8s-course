Phase 2

Planned

The Architect Track

From using the Kubernetes API to extending it, and from running one cluster to designing a fleet. Full-spectrum coverage — platform, product, and infra architecture. Operator builds in Go + kubebuilder.

The bridge

Phase 1’s through-line was “the API stores intent; something must enforce it” — NetworkPolicy needs a policy CNI, a PVC needs a provisioner, a Deployment needs the controller-manager. That “something” is a controller. An operator observes controllers; an architect writes them. Every module closes with a decision framework — architecture is judgment, not trivia.

A1 · The Controller Pattern

The heart of Kubernetes and the operator→architect moment. Control theory: desired vs observed state, level-triggered (reconcile to truth) not edge-triggered (react to events), idempotent reconcile loops, the informer/work-queue machinery.

You’ll build

First a ~20-line shell reconciler to feel the loop with zero framework, then a real CRD + controller with kubebuilder (controller-runtime) that reconciles a custom resource to reality — the same pattern kube-controller-manager uses.

Decision framework

When to reach for a controller/operator vs a Helm chart vs a Job. What belongs in an API vs config.

A2 · Extending the API

CRDs with schema validation & versioning, admission webhooks (mutating + validating), finalizers, owner references & garbage collection, the status subresource, and API aggregation. Policy-as-code with Kyverno / OPA Gatekeeper.

You’ll build

A validating admission webhook + a policy that rejects non-compliant manifests (e.g. “no BestEffort in prod,” “images must be signed”).

Decision framework

CRD vs ConfigMap vs annotation; webhook vs policy engine; how to version an API you’ll have to live with.

A3 · Scaling & Scheduling Architecture

HPA / VPA / Cluster Autoscaler / Karpenter, requests-and-limits strategy, affinity & anti-affinity, topology spread, taints/tolerations, priority & preemption, PodDisruptionBudgets, and capacity & cost modeling.

You’ll build

An autoscaling + scheduling design validated under a load test — watch pods scale and land where you designed.

Decision framework

HPA vs VPA vs both; node autoscaling (CA vs Karpenter); how to size requests without over-provisioning cost.

A4 · Platform Architecture

The big trade-off module. CNI selection (Flannel vs Calico vs Cilium/eBPF), Ingress vs Gateway API, service mesh (when it earns its complexity), and multi-tenancy: namespaces → vClusters → separate clusters, with the isolation/cost trade-off at each tier.

You’ll build

A tenancy + network design with every choice defended in writing.

Decision framework

Soft vs hard multi-tenancy; when a mesh is worth it; ingress controller selection.

A5 · Delivery & Fleet

GitOps (Argo CD / Flux), Helm vs Kustomize, progressive delivery (canary/blue-green), environment promotion, and multi-cluster fleet management.

You’ll build

A GitOps pipeline that reconciles the lab cluster from a repo — closing the loop: your cluster state becomes a controller reconciling Git.

Decision framework

Argo vs Flux; Helm vs Kustomize vs both; push vs pull delivery; how to promote across environments safely.

A6 · Reliability & Security Architecture

DR & backup (Velero, etcd strategy, multi-region), RBAC & Pod Security Admission, supply chain (image signing, SBOMs, admission gates), secrets (External Secrets / KMS), and observability architecture (metrics/logs/traces, SLOs).

You’ll build

A security posture + DR runbook for the cluster: least-privilege RBAC, enforced PSA, backup/restore drill.

Decision framework

RTO/RPO → backup strategy; where secrets live; how much supply-chain rigor a given org actually needs.

A7 · Capstone — Reference-Architecture Review

Given a realistic set of requirements (scale, compliance, budget, team shape), produce a full reference architecture — clusters, networking, tenancy, delivery, security, DR — and defend the trade-offs against my adversarial review. This is the architect’s real job: not knowing every knob, but justifying every choice.
