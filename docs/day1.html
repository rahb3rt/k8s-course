Substrate & the Control Plane

Climb from below Kubernetes up: a container built by hand from Linux primitives, a control plane assembled from four files on disk, down to the protobuf bytes in etcd — then kill it and learn what “the cluster is down” really means.

1 · A container is three Linux primitives

There is no “container” — only Linux lying to a process with three things:

Namespaces

What a process can see — pid, net, mnt, uts, ipc, user.

cgroups v2

What it can use — cpu, memory, io. The OOM killer consults these.

A pivoted rootfs

The unpacked OCI image it pivot_roots into.

That’s it

containerd just automates those three. No magic.

Lab 1.1 — build one by hand in the sandbox VM, no Docker:

sudo unshare --pid --fork --mount-proc --uts --net --ipc --mount bash
hostname isolated-box   # uts ns — host hostname unaffected
ps aux                  # pid ns — you are PID 1, you see almost nothing
ip addr                 # net ns — only 'lo', and it's DOWN
ls /                    # NOT isolated — no pivot_root, you see the whole VM
Why this matters downstream

PID 1: the kernel doesn’t apply default signal actions to PID 1, so an app ignoring SIGTERM strands pods in Terminating (Day 3). Empty netns: every pod starts exactly like that lo-only shell; the CNI wires it — no CNI → ContainerCreating (Day 2).

2 · The bootstrap paradox → static pods

The apiserver, scheduler, controller-manager, and etcd all run as pods — but pods are scheduled by the control plane that doesn’t exist yet. Resolution: static pods. The kubelet (a plain systemd service, not a pod) runs any manifest in /etc/kubernetes/manifests/ directly.

ls /etc/kubernetes/manifests/   # etcd · apiserver · controller-manager · scheduler
systemctl is-active kubelet     # active — the one thing that isn't a pod

The kubelet also creates a read-only mirror pod in the API. The file is truth: kubectl delete pod etcd-cp → it reappears instantly. And they run before the CNI because hostNetwork: true — they use the node’s network, need no pod IP. That’s why a fresh node is NotReady yet the control plane is Running.

3 · The PKI tree

FileRole
ca.crt / ca.keyCluster root CA — everything chains to it
apiserver.crtServing cert; SANs include the advertise address
apiserver-etcd-client.*Client cert to etcd — signed by the separate etcd CA
sa.key / sa.pubSigns ServiceAccount tokens — leak sa.key = forge any identity
etcd/etcd’s own CA + certs — its own trust domain

Most security-critical: sa.key. Most availability-critical: etcd/ + /var/lib/etcd.

4 · etcd is truth; the apiserver is its only interpreter

# etcdctl lives in the distroless etcd pod (no shell) — exec the binary directly.
# Pod name = etcd-<nodename>, i.e. etcd-lima-cp. Wrap the cert flags in a helper:
ectl() { kubectl -n kube-system exec -i etcd-lima-cp -- etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key "$@"; }

ectl member list -w table                          # the etcd cluster (one member)
ectl get /registry --prefix --keys-only | wc -l    # ~286 objects in an "empty" cluster
ectl get /registry/pods/kube-system/etcd-lima-cp   # binary PROTOBUF, not JSON

Objects are stored as protobuf; only the apiserver encodes/decodes it. That’s the technical reason “everything goes through the apiserver” is architecture, not etiquette. (The etcd image is distroless — no shell — so exec the etcdctl binary directly.)

Breakage lab — kill etcd, debug with kubectl dead

Stop etcd by moving its manifest out of the watched dir. The apiserver’s /readyz hits 500 and crash-loops; kubectl hangs; the worker’s workloads keep serving. Fully reversible — you stop etcd’s process, never its data.

Phase 0 · steady state (set up before you break anything)

# crictl needs its socket configured BEFORE the outage — you can't do it mid-crisis
sudo tee /etc/crictl.yaml >/dev/null <<'EOF'
runtime-endpoint: unix:///run/containerd/containerd.sock
image-endpoint: unix:///run/containerd/containerd.sock
timeout: 10
EOF
kubectl create deployment web --image=nginx --replicas=2   # a workload to test survival
kubectl get nodes && kubectl get pods -o wide            # capture the baseline
# in a 2nd cp terminal, watch the kubelet:  sudo journalctl -u kubelet -f

Phase 1 · inject

sudo mv /etc/kubernetes/manifests/etcd.yaml /root/etcd.yaml   # kubelet stops the etcd static pod

Phase 2 · observe the blast radius

sudo crictl ps | grep etcd          # gone
kubectl get nodes                   # HANGS then times out (apiserver up, etcd unreachable)
# workloads alive? check the WORKER (crictl is node-local; web runs there, cp is tainted):
limactl shell worker -- sudo crictl --runtime-endpoint unix:///run/containerd/containerd.sock ps | grep nginx

Phase 3 · diagnose (kubectl is dead — drop a layer)

sudo crictl ps -a | grep -E 'kube-apiserver|etcd'
sudo crictl logs $(sudo crictl ps -aq --name kube-apiserver | head -1) 2>&1 | tail -20   # etcd connection refused
ls -l /etc/kubernetes/manifests/    # SMOKING GUN: etcd.yaml is missing

Phase 4 · recover (self-heals)

sudo mv /root/etcd.yaml /etc/kubernetes/manifests/   # kubelet restarts etcd from the file
# wait 20-90s — apiserver crash-loops against fresh etcd, then recovers
kubectl get nodes ; kubectl get deployment web       # still 2/2 — ZERO data lost
# if apiserver stays 'refused' >2 min, kick it: bounce its manifest out and back
Lessons

Control plane down ≠ workloads down. When kubectl is dead: crictl (node-local!), journalctl -u kubelet, the manifests dir. refused = nothing listening; timeout = stuck on a dependency. Data outlives process — /var/lib/etcd survived, so recovery was lossless.

Gotchas

  • Drop --mount-proc from unshare and ps reads the host's /proc — the PID-namespace illusion breaks. That flag is load-bearing.
  • The etcd image is distroless — kubectl exec … -- sh -c fails with sh: not found. Exec the etcdctl binary directly (the ectl helper above).
  • crictl needs root (socket is 0660 root:root) and an endpoint in /etc/crictl.yaml — configure it before an outage, not during.
  • crictl is node-local: to see a workload's containers, run it on the node the pod actually runs on (cp is tainted, so web is on the worker).
  • After restoring etcd.yaml, the apiserver may show connection refused for up to ~2 min (crash-loop backoff) before recovering — expected, not a second fault. Kick it by bouncing its manifest if it's stuck longer.

Takeaways

Container = 3 primitives

Namespaces + cgroups + rootfs. Every weird-pod bug bottoms out here.

Bootstraps from disk

Static pods dodge the CNI via hostNetwork. The file is truth.

etcd holds state; apiserver gives meaning

Protobuf in, YAML out. The apiserver is etcd’s sole client.

Control plane down ≠ workloads down

refused = not listening; timeout = stuck on a dependency. Data outlives process.
