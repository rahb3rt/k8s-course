Observability: Prometheus & Grafana

Stop guessing what the cluster is doing. Deploy the kube-prometheus-stack (Prometheus + Grafana + Alertmanager + exporters) with Helm, and learn the pull-based metrics pipeline from scrape to dashboard.

The metrics pipeline (the mental model)

Prometheus is pull-based: it scrapes HTTP /metrics endpoints on a schedule and stores time series. The pieces:

ComponentJob
node-exporterPer-node DaemonSet — exposes host metrics (CPU, mem, disk, net).
kube-state-metricsTurns API objects into metrics (pod phase, restarts, deploy replicas, PVC status).
PrometheusScrapes all targets, stores TSDB, evaluates alert rules. Query with PromQL.
ServiceMonitor / PodMonitorCRDs that tell Prometheus what to scrape (a preview of Phase 2 — the API is extended with CRDs).
AlertmanagerReceives fired alerts, dedups/groups/routes them (to Slack, PagerDuty…).
GrafanaDashboards over Prometheus (and other) data sources.

5.1 · Install with Helm

curl -fsSL https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm install monitoring prometheus-community/kube-prometheus-stack \
  --namespace monitoring --create-namespace \
  --set prometheus.prometheusSpec.retention=3h

One chart deployed Prometheus + Grafana + Alertmanager + the operator + both exporters. node-exporter is a DaemonSet (one per node); kube-state-metrics is a single Deployment. The 2/2 / 3/3 READY counts are operator-injected config-reloader & dashboard sidecars.

5.2 · Reach Grafana

kubectl -n monitoring get secret monitoring-grafana -o jsonpath='{.data.admin-password}' | base64 -d
kubectl -n monitoring port-forward svc/monitoring-grafana 3000:80   # open localhost:3000 (Lima forwards it)

Pre-built Kubernetes / … and Node Exporter / Nodes dashboards, already populated. Log in as admin.

5.3 · Query with PromQL

# generate a live event to observe (crash-loops so the restart counter climbs):
kubectl apply -f manifests/oom-live.yaml
# then in Grafana → Explore (datasource: Prometheus), run PromQL:
sort_desc(kube_pod_container_status_restarts_total)      # apiserver's restart scars from Days 1 & 4
kube_pod_container_status_restarts_total{pod="oom-live"} # climbs on each refresh
kube_pod_container_status_last_terminated_reason{reason="OOMKilled"}
Observability is not retroactive

Prometheus only knows what it scraped since it started. Day-3 events that predate it are gone. Deploy monitoring before the incident — a metric you weren't collecting is a question you can never answer.

5.4 · ServiceMonitors, PrometheusRules & a firing alert

Prometheus discovers targets via ServiceMonitor CRDs; alerts are PrometheusRule CRDs. The Prometheus Operator — a controller — watches both and reconciles them into running config. That's the exact pattern you build in Phase 2 A1. We wrote a rule that fired on a self-inflicted OOM crash-loop.

kubectl get servicemonitor -A       # ~a dozen — one per component
kubectl apply -f manifests/oom-live-alert.prometheusrule.yaml
ALERTS{alertname="OomLiveCrashLooping"}   # in Grafana Explore: pending -> firing -> Alertmanager

Gotchas

  • The stack is memory-hungry. On a small worker, cap Prometheus (--set prometheus.prometheusSpec.resources.requests.memory=300Mi) or pods land Pending/OOMKilled (a Day-3 callback).
  • PrometheusRule and ServiceMonitor need the release: monitoring label — the operator's selector ignores anything without it, silently.
  • Prometheus here uses emptyDir — all metrics vanish on pod restart/reschedule. Give it a PVC in production.
  • Observability is not retroactive: no data exists for anything before Prometheus started scraping.
  • port-forward binds the VM's localhost; Lima forwards it to your Mac's localhost:3000. Grafana login: user admin, password from the monitoring-grafana secret.
  • Alerts have a for: delay — the condition must hold that long before the alert flips pending → firing.

Takeaways

Pull-based pipeline

Prometheus scrapes /metrics; exporters expose them; Grafana draws; Alertmanager routes.

Operators reconcile CRDs

ServiceMonitor/PrometheusRule are intent; the operator makes them real — a live preview of Phase 2.

Monitor before the incident

There is no historical data for what you weren't scraping.

Helm = platform delivery

Six components, one chart, dozens of objects — how real platforms ship.
