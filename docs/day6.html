Cluster Upgrades

The real, hands-on version of Day 4.3: take this cluster from one minor version to the next — control plane first, then each node — without dropping the workloads.

The rules (version skew) — never break these

  • One minor at a time. 1.31 → 1.32, never 1.31 → 1.33. To go up two minors you upgrade twice.
  • Control plane leads, nodes follow. The API server must be on the new version before any node.
  • kubelet may lag the API server by ≤ 3 minors, never lead it. This is what makes a mixed-version cluster (mid-upgrade) valid.
  • Drain before you upgrade a node. The Day-4.3 drill is literally step one of every node upgrade.

Step 0 · Pre-flight (skip these and you get bitten)

The upgrade command is the easy part. The failures happen because of what you didn't check first.

Scan for removed APIs

Each minor removes old API versions. In a big cluster this silently breaks manifests/charts. Scan first: pluto detect-all-in-cluster or kubent, and fix every hit before upgrading. The #1 cause of "the upgrade broke half our services."

Read the release notes

Skim the target version's changelog for removals, deprecations, and behavioral changes. Two minutes that saves an outage.

Back up etcd

Day 4.1 — always snapshot before touching the control plane. etcdctl snapshot save + verify.

Check readiness

All nodes Ready, no pods pending, PDBs sane, and enough spare capacity for drains to reschedule (Day 4.3).

sudo cp -r /etc/kubernetes/pki /root/pki-backup-$(date +%s)   # cheap insurance
kubectl -n kube-system exec -i etcd-lima-cp -- etcdctl \
  --endpoints=https://127.0.0.1:2379 --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /var/lib/etcd/pre-upgrade.db && sudo cp /var/lib/etcd/pre-upgrade.db /root/

Step 1 · Control plane (on cp)

Upgrade the kubeadm tool first — it's what drives everything else. Then plan, apply, and finally the kubelet.

# 1a. Repoint the package repo to the new minor, then upgrade the kubeadm binary
sudo sed -i 's|/v1.31/|/v1.32/|' /etc/apt/sources.list.d/kubernetes.list
sudo apt-get update
apt-cache madison kubeadm | head          # find the exact patch, e.g. 1.32.13-1.1
sudo apt-mark unhold kubeadm
sudo apt-get install -y kubeadm='1.32.*'
sudo apt-mark hold kubeadm
kubeadm version                            # confirm v1.32.x before continuing

# 1b. Plan — shows current vs target and every component it will touch. READ this table.
sudo kubeadm upgrade plan

# 1c. Apply — use the EXACT version string (not a placeholder). This is the big step.
sudo kubeadm upgrade apply v1.32.13 -y
What upgrade apply actually does

It pulls the new control-plane images and swaps the static pods one at a time — etcd → apiserver → controller-manager → scheduler — writing each new manifest into /etc/kubernetes/manifests/ and backing up the old one to /etc/kubernetes/tmp/kubeadm-backup-manifests-* (your rollback). It also renews every certificate as a side effect (Day 4.2 for free). Ends with SUCCESS! A control plane node ... was upgraded.

# 1d. Upgrade this node's kubelet + kubectl, then restart the kubelet
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet='1.32.*' kubectl='1.32.*'
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload && sudo systemctl restart kubelet
kubectl get nodes     # cp -> v1.32.13; worker still 1.31 (a VALID mixed state)
HA control plane (3+ cp nodes)

Run kubeadm upgrade apply on the first control-plane node only. On the other cp nodes run kubeadm upgrade node instead (no apply), then upgrade each one's kubelet. Do them one at a time so quorum never drops.

Step 2 · Each worker (repeat / batch across the fleet)

Now the nodes, one at a time (or in small batches, PDBs permitting). Every worker follows the same four moves:

# 2a. on the WORKER — repoint repo + upgrade kubeadm, then its local config
sudo sed -i 's|/v1.31/|/v1.32/|' /etc/apt/sources.list.d/kubernetes.list
sudo apt-get update
sudo apt-mark unhold kubeadm && sudo apt-get install -y kubeadm='1.32.*' && sudo apt-mark hold kubeadm
sudo kubeadm upgrade node          # upgrades this node's kubelet config + certs

# 2b. on CP — drain the node so its pods move off (respects graceful shutdown + PDBs)
kubectl drain lima-worker --ignore-daemonsets --delete-emptydir-data
#     single-worker lab -> pods go Pending until uncordon. Multi-worker -> they reschedule, zero downtime.

# 2c. on the WORKER — upgrade the kubelet + kubectl, restart
sudo apt-mark unhold kubelet kubectl && sudo apt-get install -y kubelet='1.32.*' kubectl='1.32.*' && sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload && sudo systemctl restart kubelet

# 2d. on CP — return the node to service
kubectl uncordon lima-worker

Step 3 · Verify & rollback

kubectl get nodes                    # EVERY node reports v1.32.13, all Ready
kubectl get pods -A -o wide          # everything rescheduled and Running
Rollback

Control-plane pods: restore from /etc/kubernetes/tmp/kubeadm-backup-manifests-*. Whole cluster: the etcd snapshot from Step 0 (Day 4.1). Nodes: cordon the bad one, shift workloads off, replace it.

At fleet scale, don't do this by hand

100 nodes → managed node pools (EKS/GKE/AKS, surge upgrades) or Cluster API (a MachineDeployment rolls nodes like a Deployment rolls pods). Blue-green new nodes, drain old, delete. See the fleet-upgrade Field Note.

Gotchas

  • upgrade plan will advertise newer minors (e.g. 1.36) — ignore them, target only the next minor.
  • The renewal/upgrade is a no-op until the static pods restart — files new, process old (same trap as cert rotation).
  • A stuck drain = a PodDisruptionBudget you can't satisfy → check kubectl get pdb -A and your capacity (Day 4.3).
  • On a cluster only hours old the cert expiry year won't visibly change on renewal — the timestamp does.
  • Node-local PVs (our local-path) don't follow a pod to a new node — network-attached storage does.
