Cluster Upgrades

The real, hands-on version of Day 4.3: take this cluster from one minor version to the next — control plane first, then each node — without dropping the workloads.

The rules (version skew)

  • One minor at a time. 1.31 → 1.32, never 1.31 → 1.33.
  • Control plane leads, nodes follow. Upgrade the control plane first.
  • kubelet may lag the API server by ≤ 3 minors, never lead it.
  • Drain before you upgrade a node — the Day-4.3 drill is step one of every node upgrade.

The upgrade, done (1.31.14 → 1.32.13)

Control plane (cp)

sudo sed -i 's|/v1.31/|/v1.32/|' /etc/apt/sources.list.d/kubernetes.list ; sudo apt-get update
sudo apt-mark unhold kubeadm && sudo apt-get install -y kubeadm='1.32.*' && sudo apt-mark hold kubeadm
sudo kubeadm upgrade plan
sudo kubeadm upgrade apply v1.32.13 -y          # swaps static pods + renews all certs
sudo apt-get install -y kubelet='1.32.*' kubectl='1.32.*' ; sudo systemctl restart kubelet

Then each worker

# on worker: upgrade kubeadm ; sudo kubeadm upgrade node
# on cp:     kubectl drain lima-worker --ignore-daemonsets --delete-emptydir-data
# on worker: upgrade kubelet ; sudo systemctl restart kubelet
# on cp:     kubectl uncordon lima-worker ; kubectl get nodes   # ALL v1.32.13
Mixed-version is valid

Mid-upgrade, cp ran 1.32 while the worker ran 1.31 — the skew rule (kubelet ≤3 minors behind, never ahead) is what makes rolling upgrades possible.

Capacity buys safety

Draining the only worker = a blip. ≥2 workers = zero downtime (pods reschedule). Same lesson as the Day-4.3 drain. Prometheus on emptyDir lost its data → prod needs a PVC.

Rollback: pre-upgrade manifests in /etc/kubernetes/tmp/kubeadm-backup-manifests-*. Runbook: day6-cluster-upgrade/upgrade-runbook.md.
