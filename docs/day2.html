Networking & the Dataplane

How a 10.244.x.x pod packet crosses a 192.168.104.0/24 link the VMs don’t route pod IPs on — traced end to end, then broken and fixed by hand.

1 · The model: IP-per-pod flat network

Every CNI must satisfy: every pod gets a real, unique, routable IP; every pod reaches every other pod directly, no NAT; nodes can reach all pods. Two implementations:

Same-node

A Linux bridge (cni0) + a veth pair per pod — a virtual cable, one end in the bridge, the other is the pod’s eth0. Pure L2.

Cross-node

A VXLAN overlay (flannel.1). The pod packet is wrapped in a node-to-node UDP packet.

The routing table is the decision:

10.244.1.0/24 dev cni0                          # this node's pods → bridge (local)
10.244.0.0/24 via 10.244.0.0 dev flannel.1 ...  # other node's pods → VXLAN tunnel

Map it on your cluster

kubectl apply -f manifests/netlab.yaml   # busybox test pod (ping/wget/nslookup)
kubectl get pods -o wide                 # note pod IPs + which node each is on
# on the WORKER:
ip -brief addr    # cni0 (bridge) · flannel.1 (VXLAN) · a veth@ifN per pod
ip route          # the local-vs-overlay decision, in two lines
sudo cat /etc/cni/net.d/10-flannel.conflist   # the CNI plugin chain (flannel → bridge → portmap)

2 · Trace the packet (two layers, one ping)

# Worker term 1 — the WIRE (encapsulated):  sudo tcpdump -ni eth0 udp port 8472
# Worker term 2 — the TUNNEL exit (decap):   sudo tcpdump -ni flannel.1 icmp
kubectl exec netlab -- ping -c3 <coredns-pod-IP-on-cp>

flannel.1 shows the inner 10.244→10.244 ICMP; eth0 shows the same packet wrapped as 192.168.104.3 → .1 : UDP/8472 (tcpdump mislabels 8472 as “OTV” — it’s VXLAN). Overhead = 50 bytes → pod MTU is 1450. “Small requests work, big ones hang” = suspect MTU.

3 · A Service is just iptables rules

kubectl expose deployment web --port=80
sudo iptables-save -t nat | grep <clusterIP>       # KUBE-SERVICES → KUBE-SVC-xxxx
sudo iptables-save -t nat | grep KUBE-SVC-xxxx      # probability chain → KUBE-SEP (DNAT)

The ClusterIP exists nowhere as an interface — only as a match target. KUBE-SVC load-balances by statistic mode random probability (biased coin flips that sum to fair shares); each KUBE-SEP does the DNAT to a real pod; conntrack pins the pick. KUBE-MARK-MASQ SNATs off-pod-network sources so replies return.

Breakage (self-driven) — the vanishing replies

kubectl exec netlab -- ping -c3 -W2 <cp-pod-IP>       # cross-node → 100% loss

sudo tcpdump -ni eth0 udp port 8472   ;   sudo tcpdump -ni flannel.1 icmp
# inspect + fix (worker):
sudo iptables -L INPUT -n -v --line-numbers          # find the DROP udp dpt:8472 (climbing counter)
sudo iptables -D INPUT -p udp --dport 8472 -j DROP   # fix
kubectl exec netlab -- nslookup kubernetes.default.svc.cluster.local  # verify the REAL symptom (DNS)
Real-world signature

Same-node OK + cross-node broken + DNS timeouts = a firewall / security group blocking the overlay port (8472 Flannel VXLAN, 4789 std VXLAN, 179 Calico BGP).

NetworkPolicy does nothing (under Flannel)

kubectl apply -f manifests/deny-web.networkpolicy.yaml
kubectl exec netlab -- wget -qO- --timeout=3 http://<web-pod-IP>   # STILL works

A NetworkPolicy is intent; enforcement is delegated to the CNI. Flannel has no policy engine, so it’s accepted, stored, and silently ignored — fails open. Enforce with a policy CNI (Calico/Cilium) and always test a deny.

Gotchas

  • tcpdump labels UDP/8472 as "OTV" — it is Flannel VXLAN, don't be thrown. Standard VXLAN is 4789; Flannel uses the older 8472.
  • tcpdump taps the wire before netfilter — a packet visible on eth0 but never on flannel.1 was dropped in the INPUT chain. That gap is the diagnosis.
  • busybox uses musl, whose resolver doesn't walk search domains — short names like kubernetes.default return NXDOMAIN. Test with the FQDN. (A response — even NXDOMAIN — means DNS works; a hang means it's dead.)
  • Overlapping pod and node CIDRs = silent routing breakage. Keep pod CIDR (10.244/16) disjoint from the node net (192.168.104/24).
  • A NetworkPolicy under Flannel is silently unenforced (fails open, no error). Always test a deny before trusting it.
  • kube-proxy needs conntrack (a fatal kubeadm preflight) and bridge-nf-call-iptables=1, or Services "exist" but no traffic reaches a pod.

Takeaways

Flat pod IPs, two mechanisms

Same-node = bridge, cross-node = VXLAN; the routing table chooses. 50-byte overhead → MTU 1450.

A Service is a NAT illusion

iptables DNAT + conntrack, not a process.

Response ≠ hang

NXDOMAIN means DNS works; a timeout means dead. ndots:5 makes short names slow — prefer FQDNs.

Intent needs an enforcer

NetworkPolicy ⇒ a policy CNI. “Security object exists” ≠ “security enforced.”
