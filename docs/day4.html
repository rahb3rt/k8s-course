Production Operations

The drills that separate people who run clusters from people who deploy to them: resurrect a dead cluster from a snapshot, rotate expired certs, drain safely under a PDB, and clear a multi-fault incident cold.

4.1 · etcd backup & restore

Done

The dark twin of Day 1’s breakage. There you stopped etcd’s process and lost nothing. Here you destroyed its data (rm -rf /var/lib/etcd) and rebuilt the whole cluster from a snapshot — measured RTO ≈ 2 min.

# 1. plant a canary, take a VERIFIED snapshot, move it to safety
kubectl create configmap canary-before --from-literal=when=before
kubectl -n kube-system exec -i etcd-lima-cp -- etcdctl \
  --endpoints=https://127.0.0.1:2379 --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /var/lib/etcd/backup.db
sudo cp /var/lib/etcd/backup.db /root/etcd-backup.db
kubectl create configmap canary-after --from-literal=when=after   # doomed: created after the snapshot

# 2. DISASTER: stop the control plane, destroy the data
sudo mv /etc/kubernetes/manifests /etc/kubernetes/manifests.off   # kubelet stops static pods
sudo rm -rf /var/lib/etcd

# 3. RESTORE (etcdutl on the host; flags MUST match the running etcd)
sudo etcdutl snapshot restore /root/etcd-backup.db \
  --name=lima-cp --initial-cluster=lima-cp=https://192.168.104.1:2380 \
  --initial-advertise-peer-urls=https://192.168.104.1:2380 --data-dir=/var/lib/etcd

# 4. REVIVE + verify — canary-before survives, canary-after is GONE (= your RPO)
sudo mv /etc/kubernetes/manifests.off /etc/kubernetes/manifests
kubectl get nodes ; kubectl get cm
RPO — the canary test

A ConfigMap created after the snapshot vanished on restore. Data since the last backup = lost. Backup frequency is a business decision.

Defense in depth

Odd-quorum etcd (3/5) survives node loss — but replication spreads corruption instantly. So: quorum + frequent verified snapshots + a tested restore runbook.

Full runbook: day4-production-ops/labs/4.1-etcd-backup-restore.md · cron-able backup: etcd-backup.sh

4.2 · Certificate expiry & rotation

Done

kubeadm leaf certs expire in 1 year; the CAs last ~10. A cluster left un-upgraded for a year locks itself out — x509: certificate has expired. Rotation re-signs the leaves with the same CAs, so nothing else has to change.

sudo kubeadm certs check-expiration    # leaf ~364d · CA ~9-10y
sudo kubeadm certs renew all           # reissue every leaf
# GOTCHA: restart the static pods — the running apiserver holds the OLD cert in memory
sudo mv /etc/kubernetes/manifests{,.off} && sleep 20 && sudo mv /etc/kubernetes/manifests{.off,}
sudo cp /etc/kubernetes/admin.conf $HOME/.kube/config   # refresh your kubeconfig too
Architect's insight

kubeadm upgrade renews these certs as a side effect — so “upgrade at least yearly” is your rotation mechanism. Un-upgraded clusters are cert time-bombs. Full runbook: labs/4.2-cert-rotation.md.

4.3 · Upgrades, drain, cordon & PDBs

Done

Cordon (unschedulable) → drain (evict) → PDB (availability contract the drain must honor). Watched a minAvailable: 2 PDB block a drain: 2 replicas, 1 worker, nowhere to reschedule → the drain correctly refuses.

kubectl cordon lima-worker
kubectl drain lima-worker --ignore-daemonsets --delete-emptydir-data
# -> "Cannot evict pod as it would violate the pod's disruption budget"
kubeadm upgrade plan ; kubeadm upgrade apply v1.31.x   # control plane first, one minor at a time
Design rule

PDB + spare capacity + spread across nodes = safe rolling upgrades. PDB without capacity = a node that won’t drain. kubelet may lag the apiserver ≤3 minors, never lead. Runbook: labs/4.3-drain-pdb-upgrades.md.

4.4 · Capstone — multi-fault incident

Done

Two independent faults injected at once, diagnosed cold: a Deployment stuck Pending (100Gi request → FailedScheduling, a wedged rollout) and a silent DNS outage (CoreDNS scaled to 0). Triage → separate the problems → root-cause each → fix → verify the real symptom.

# inject (blind): an unschedulable Deployment + CoreDNS to zero
kubectl create deployment shop --image=nginx:1.27 --replicas=2
kubectl set resources deployment shop --requests=memory=100Gi,cpu=32
kubectl -n kube-system scale deployment coredns --replicas=0
# triage → diagnose:
kubectl get pods -A                               # shop-* Pending; coredns MISSING
kubectl describe pod <shop-pending-pod> | tail    # FailedScheduling: Insufficient cpu/memory
kubectl exec sts-0 -- nslookup kubernetes.default.svc.cluster.local   # DNS down
# fix both, then verify the real symptoms:
kubectl set resources deployment shop --requests=memory=64Mi,cpu=100m
kubectl -n kube-system scale deployment coredns --replicas=2
What closed it

Triage separated the two problems; “what’s missing” caught the silent CoreDNS fault; the FailedScheduling event handed over the scheduling cause; response-vs-hang and verify-the-symptom confirmed both fixes. Scripts: labs/4.4-capstone-break.sh / -reset.sh.

Optional side-quest

Install Calico policy enforcement so the Day-2 deny-web NetworkPolicy actually bites — watch the same wget that succeeded under Flannel start timing out.

Gotchas

  • etcd restore: --name/--initial-cluster/--initial-advertise-peer-urls must match the running etcd, or the member won't rejoin (cluster-ID mismatch). And etcdutl refuses to restore over an existing data dir — rm -rf /var/lib/etcd first.
  • Cert rotation is a no-op until you restart the static pods (files new, process old) and copy admin.conf back into ~/.kube/config. On a young cluster the expiry year won't change — only the timestamp does.
  • A drain that stalls = a PodDisruptionBudget you can't satisfy → kubectl get pdb -A and check capacity. Draining also permanently deletes bare pods (no controller to recreate them); Deployment/StatefulSet pods come back.
  • kubeadm upgrade apply needs the exact version (v1.32.13), not a placeholder like v1.32.X.
  • The apiserver crash-loops briefly after any control-plane bounce (etcd restart, cert rotation, upgrade) — retry kubectl, it's expected.
