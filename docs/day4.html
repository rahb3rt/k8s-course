The drills that separate people who run clusters from people who deploy to them. This page fills in with commands as we do each lab.

4.1 · etcd backup & restore

Done

The dark twin of Day 1’s breakage. There you stopped etcd’s process and lost nothing. Here you destroyed its data (rm -rf /var/lib/etcd) and rebuilt the whole cluster from a snapshot — measured RTO ≈ 2 min.

etcdctl snapshot save backup.db                # verified with: etcdutl snapshot status
sudo etcdutl snapshot restore backup.db \      # flags MUST match the running etcd:
  --name=lima-cp \
  --initial-cluster=lima-cp=https://192.168.104.1:2380 \
  --initial-advertise-peer-urls=https://192.168.104.1:2380 \
  --data-dir=/var/lib/etcd
RPO — the canary test

A ConfigMap created after the snapshot vanished on restore. Data since the last backup = lost. Backup frequency is a business decision.

Defense in depth

Odd-quorum etcd (3/5) survives node loss — but replication spreads corruption instantly. So: quorum + frequent verified snapshots + a tested restore runbook.

Full runbook: day4-production-ops/labs/4.1-etcd-backup-restore.md · cron-able backup: etcd-backup.sh

4.2 · Certificate expiry & rotation

Deliberately expire an API server cert (from the PKI tree toured on Day 1), watch the cluster lock itself out, then rotate.

kubeadm certs check-expiration
kubeadm certs renew all        # then restart control-plane static pods + refresh kubeconfigs

4.3 · Upgrades, drain, cordon & PDBs

Move workloads off a node safely and roll a version bump — and see how a PodDisruptionBudget can block a drain.

kubectl cordon <node> ; kubectl drain <node> --ignore-daemonsets
kubeadm upgrade plan ; kubeadm upgrade apply v1.31.x
