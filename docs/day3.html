Storage, Lifecycle & the OOM Killer

State and time: how pods are torn down, who the kernel kills under memory pressure, and how data outlives the pod that wrote it.

1 · How a pod dies

Graceful shutdown sequence on kubectl delete:

  1. Pod gets a deletionTimestamp → Terminating.
  2. In parallel: removed from Service endpoints and shutdown begins.
  3. preStop hook runs (if any).
  4. Runtime sends SIGTERM to PID 1.
  5. Kubelet waits up to terminationGracePeriodSeconds (default 30).
  6. Still alive? SIGKILL.
kubectl apply -f manifests/stubborn-pod.yaml   # PID 1 ignores SIGTERM (trap "" TERM)
time kubectl delete pod stubborn               # ~30s — waited out the grace period
The nastier cousin

Endpoint removal races shutdown. A fast-exiting app can be dead while some node’s iptables still lists it → 502s mid-rollout. Fix: a preStop sleep (~5–10s) so it keeps serving during the drain window.

2 · QoS & the OOM killer

QoS class is derived from requests/limits and becomes an oom_score_adj the kernel reads:

QoSRuleoom_score_adjKilled
Guaranteedrequests == limits-997last (near-immune)
Burstablesome requests/limits2–999middle
BestEffortnothing set1000first
 
kubectl apply -f manifests/qos-pods.yaml
kubectl exec qos-guaranteed -- cat /proc/1/oom_score_adj   # -997
kubectl apply -f manifests/oom-victim.yaml                 # asks 300Mi under a 128Mi limit
kubectl get pod oom-victim -w                              # OOMKilled, exit 137 (=128+9)
Per-container OOM

Exceeded your own limits.memory → deterministic cgroup kill, QoS-irrelevant. Raise the limit or fix the leak.

Node-level OOM

Node is full → kubelet evicts by QoS (BestEffort first) before the kernel kills by oom_score_adj.

3 · Storage — a claim needs an actor

Chain: Pod → PVC (claim) → PV (volume) → StorageClass → provisioner. Same pattern as NetworkPolicy — a claim is inert until a provisioner exists.

kubectl apply -f manifests/pvc-data.yaml
kubectl get pvc data          # Pending — no provisioner yet
# install local-path (see manifests/local-path-provisioner.md), then:
kubectl apply -f manifests/writer-pod.yaml   # a consumer → triggers WaitForFirstConsumer binding
kubectl delete pod writer
kubectl apply -f manifests/reader-pod.yaml   # NEW pod reads the note → data survived
Reclaim-policy footgun

kubectl get pv shows Delete (default for dynamic volumes) → deleting the PVC destroys the data. For databases, use Retain.

4 · StatefulSets — stable identity

Deployments give anonymous cattle. Databases need ordinal names, per-pod private storage, ordered lifecycle, and per-pod DNS (headless Service).

kubectl apply -f manifests/statefulset-sts.yaml
kubectl get pods -l app=sts -w   # sts-0 Ready BEFORE sts-1 (ordered)
kubectl delete pod sts-0
kubectl logs sts-0               # boot.log has TWO lines — same name, same volume

Takeaways

Termination is a deadline

SIGTERM → grace → SIGKILL. Watch the endpoint-removal race.

Two OOMs

Per-container limit (deterministic) vs node-level (QoS-ordered). Set requests.

A claim needs a provisioner

WaitForFirstConsumer delays binding to a consumer.

StatefulSets = stable identity

Ordinal names, private PVCs, ordered lifecycle, per-pod DNS.
